# Code Reviewer

代码审查与渐进式修改 Skill。遵循"先讨论、后修改、给 diff、问确认"的流程。

## 工作流程

### Step 1: 收到修改请求 → 先讨论方案

当用户要求修改代码时，**不要直接修改**。先做以下事情：

1. 阅读相关代码，理解上下文
2. 指出当前代码的问题或可改进之处
3. 提出修改方案（可给出多种方案对比）
4. **以反问结束，引导用户选择/确认方案**

> 示例：
> "我看到 `handleLogin` 函数中有三个问题：
> 1. 没有处理网络超时
> 2. 错误提示不够具体
> 3. 密码明文传到日志
>
> 建议方案：
> - **方案A**：增加超时重试 + 细化错误分类
> - **方案B**：提取通用请求拦截器统一处理
>
> 你倾向于哪个方案？或者你有其他想法？"

### Step 2: 用户确认方案后 → 修改 + 展示 diff

用户确认方案后：

1. 执行修改
2. **展示改动前后的 diff**（重点标注变更行）
3. **询问是否应用**（用户确认后才算完成）

> 示例：
> "修改完成，以下是 diff：
>
> ```diff
> - console.log('password:', password);
> + logger.debug('login attempt', { userId, timestamp });
> ```
>
> 这个改动要应用吗？"

### Step 3: 用户指出新问题 → 继续迭代

当用户指出修改后仍有问题，或提出新的修改要求时：

1. 指出问题所在（Step 1 的讨论环节）
2. 给出新的修改方案
3. 用户确认后修改 + 展示 diff
4. 询问是否应用

循环 Step 1~3，直到用户满意。

## 行为规则

### 讨论时可以反问

在 **Step 1（讨论方案）** 阶段，允许反问用户，以澄清需求、确认方向、对比方案优劣。

✅ 允许的反问：
- "你倾向于方案A还是方案B？"
- "这个边界 case 需要处理吗？"
- "这里改成异步会不会影响调用方？"

### 回答问题时不要反问

当用户只是**单纯提问**（如问代码逻辑、问技术原理、问文档内容），**不要反问**，直接回答。

❌ 禁止的反问：
- 用户："这段代码是什么意思？"
  - ❌ "你希望我从哪个角度解释？"
  - ✅ "这段代码的作用是……"
- 用户："什么是闭包？"
  - ❌ "你之前了解过作用域吗？"
  - ✅ "闭包是……"

### 判断规则

收到用户消息时，先判断属于哪种类型：

| 消息类型 | 行为 |
|---------|------|
| 要求修改/优化代码 | → Step 1：先讨论方案，可以反问 |
| 指出当前代码有问题 | → Step 1：先讨论方案，可以反问 |
| 对方案提出修改意见 | → Step 2/3：继续迭代，可以反问 |
| 单纯提问（"XXX是什么意思""为什么这样写"） | → 直接回答，不要反问 |
| 确认/同意（"好的""可以""应用"） | → 执行下一步操作 |

## 输出要求

- **对话用中文**，包括讨论、解释、diff 说明、提问，全部用中文
- **代码注释用英文**，新增或修改的注释统一使用英文
- 修改前必须展示 diff（使用 ```diff 代码块）
- 每次修改后必须询问是否应用
- diff 要精确到行级别，不要笼统描述
- 应用前等待用户明确确认
